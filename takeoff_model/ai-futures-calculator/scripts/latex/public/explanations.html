2 Inputs

The quantities below are provided to the model as input time series, i.e. they progress the same way across time regardless of the rest of the model simulation (this is often referred to as the quantities being exogenous).

2.1 Human coding labor

Human coding labor at time t , L C , H ⁢ ( t ) , is the number of full-time-equivalent human coders at time t at the leading AGI company. The main task we have in mind for coders is the implementation of experiments for AI R&D.

Accounting for coders of different skill levels

Since our model converts AI coding labor into human-coder equivalents, and not all humans have the same level of coding skill, formally L C , H ⁢ ( t ) will be expressed in terms of what we call “clone-parity coders”. This is an operationalization of a typical human coder: if every human coder at time t were replaced with a clone-parity coder, productivity remains unchanged.

One could also imagine normalizing based on wages.

For simplicity our current input time series does not attempt to adjust for quality.

2.2 Automation compute

Automation compute at time t , C aut ⁢ ( t ) , measures the compute used for running coding agents that help with experiment implementation. These coding agents can either augment human engineers or replace them entirely.

C aut ⁢ ( t ) is estimated by forecasting time series data for compute growth at frontier AI companies, then estimating that coding automation compute accounts for 5% of this total. This is likely an underestimate today, but 5 % is our best guess for the automation compute fraction as we approach and attain full coding automation, when automated coders matter a lot more than today.

2.3 Experiment compute

Experiment compute at time t , C xpm ⁢ ( t ) , measures the compute used for research experiments.

C xpm ⁢ ( t ) is estimated by forecasting time series data for the compute growth at frontier AI companies, then estimating that experiment compute accounts for 50% of this total from present day onward, while covering as much as 75% in the past.

2.4 Human experiment selection skill

Experiment selection skill (called “research taste” in our prior work) measures the value one gets per experiment. We take an expansive definition of experiment selection, including: high-level research direction setting, low-level choices, and experiment ideation and interpretation.

We model the distribution T H of human researcher experiment selection skill with a lognormal distribution of mean 1. We choose the standard deviation of the lognormal so that the experiment selection skill of the 99.9 th percentile human researcher exceeds that of the median researcher by a factor of T var . We conducted surveys of frontier AI researchers to inform the value of T var .

Why use a lognormal distribution for experiment selection skill?

We choose a lognormal distribution because based on surveys of and discussions with frontier AI researchers, it seems likely that the top few percent of researchers provide much more value via experiment selection than typical researchers. Additioanlly, analogous quantities like scientific citation counts or startup success are often empirically lognormal.

2.5 Training compute

Training compute, C train ⁢ ( t ) , measures the compute used to train the most capable model available at time t . The units of C train ⁢ ( t ) are 16-bit floating-point operations (FLOP).

C train ⁢ ( t ) is estimated by extrapolating past trends, then predicting a slowdown in the late 2020s due to hitting investment limits.

3 Aggregate Labor

3.1 Aggregate coding labor

Aggregate coding labor, L C ⁢ ( t ) , is a measure of the total coding labor accounting for both human coders and AI coders.

We use indices i in the interval [ 0 , 1 ] to model the various coding tasks involved in implementing AI experiments. The AI project allocates human coding labor L C , H , i ⁢ ( t ) and AI coding labor C aut , i ⁢ ( t ) to task i . The aggregate coding labor for task i is then

G i ⁢ ( t ) = L C , H , i ⁢ ( t ) + η i ⁢ ( t ) ⋅ C aut , i ⁢ ( t ) .

Here η i ⁢ ( t ) is the conversion rate between units of automation compute and human coders (see the section about Coding Automation Fraction and Efficiency).

Since there are diminishing returns to increasing coding labor only in certain types of coding tasks without increasing coding labor in others, the aggregate coding labor L C ⁢ ( t ) is naturally modeled with a CES (“constant elasticity of substitution”) production function of the coding labors G i ⁢ ( t ) for each task:

L C ⁢ ( t ) = ( ∫ 0 1 G i ρ C ⁢ ( t ) ⁢ 𝑑 i ) 1 / ρ C = ( ∫ 0 1 ( L C , H , i ⁢ ( t ) + η i ⁢ ( t ) ⋅ C aut , i ⁢ ( t ) ) ρ C ⁢ 𝑑 i ) 1 / ρ C , ρ C < 0 .

If one increases coding labor only in certain types of coding tasks but not others, then the coding will bottleneck on those more neglected tasks.

There are diminishing returns to only increasing the coding labor for some of the labor-units, but not the others. In particular,

In order to combine automated coding labor with human coding labor, we use a CES that with parameter ρ C controls the level of bottlenecking. In particular, it sets to what extent coding is a fixed list of tasks that must always be done i.e. Amdahl’s law, vs. to what extent one can do more of the tasks that are automated in order to “substitute” for non-automated tasks. The CES currently assumes that humans can seamlessly re-allocate across tasks, which is not fully accurate. This could potentially be hackily accounted for by adjusting lowering ρ C .

Experiment throughput goes up with both experiment compute C xpm ⁢ ( t ) and coding labor L C ⁢ ( t ) , but there are diminishing returns to only increasing one of the inputs and not the other. In particular, if we take one of C xpm ⁢ ( t ) and L C ⁢ ( t ) to infinity but hold the other input fixed, then experiment throughput should remain bounded. Indeed, if you have infinite coding labor but only a fixed amount of experiment compute, you can only implement so many experiments per unit time; and if you have infinite experiment compute but only a fixed number of coders, you don’t have enough workers to code up experiments that (usefully) use the compute. This situation comes up in economics, where we would say that experiment compute and coding labor are complementary inputs that are hard to substitute for each other. It’s well modeled with a CES (“constant elasticity of substitution”) production function with substitution parameter ρ X < 0 :

X ⁢ ( t ) = ( α ⁢ C ~ xpm ⁢ ( t ) ρ X + ( 1 − α ) ⁢ L ~ C ⁢ ( t ) ρ X ) 1 / ρ X , 0 < α < 1 , ρ X < 0 ,

where

C ~ xpm ⁢ ( t ) = C xpm ⁢ ( t ) ζ , L ~ C ⁢ ( t ) = L C ⁢ ( t ) λ , 0 < ζ < 1 ,  0 < λ < 1 .

What is ρ ? The smaller the substitution parameter ρ is, the less it’s possible to only increase one of C xpm ⁢ ( t ) and L C ⁢ ( t ) in order to increase X ⁢ ( t ) .

When running simulations of our model, at each timestep we set L C , H , i ⁢ ( t ) and C aut , i ⁢ ( t ) to maximize the aggregate coding labor L C ⁢ ( t ) (subject to the constraints that the allocations L C , H , i sum to the total human coding labor L C , H ⁢ ( t ) and the allocations C aut , i ⁢ ( t ) sum to the total automation compute C aut ⁢ ( t ) ). This corresponds to the AI project allocating its human and AI coding workforce to where it’s most useful; more concretely, the human coders will mostly be reallocated to tasks that still evade automation while the AI coders will be allocated to the tasks they can efficiently automate (especially those tasks they can automate very quickly).

3.2 Aggregate experiment selection skill

Aggregate experiment selection skill, T ⁢ ( t ) , is the experiment selection skill (roughly, value per experiment) that results from combining that of AIs and humans. T H is a lognormal that models human experiment selection skill, while T A ⁢ I ⁢ ( t ) represents the AIs’ experiment selection skill at time t ( T H is a distribution but T A ⁢ I ⁢ ( t ) is just a number). We assume that once a human has a lower skill level than the AIs, the AIs replaces them, i.e. for each human their skill is replaced with

max ⁡ ( T H , T A ⁢ I ⁢ ( t ) ) .

The aggregate experiment selection skill at time t is the average of this new distribution:

T ⁢ ( t ) = 𝔼 ⁢ [ max ⁡ ( T H , T A ⁢ I ⁢ ( t ) ) ] .

We currently do not model the labor needed to do experiment selection, and believe this doesn’t drastically change results since for experiment selection, quality matters much more than quantity.

4 Software R&D

4.1 Experiment throughput

Experiment throughput, X ⁢ ( t ) , measures the AI project’s capacity for implementing and running experiments, and can roughly be understood as the number of experiments the project can run per unit time, where experiments are weighted by how much compute they use and how labor-intensive they are to code. It corresponds to how fast the AI project is going (i.e. research effort R ⁢ E ⁢ ( t ) ) if we hold the aggregate experiment selection skill T ⁢ ( t ) fixed. See the section on research effort for more explanation.

Experiment throughput goes up with both experiment compute C xpm ⁢ ( t ) and coding labor L C ⁢ ( t ) , but there are diminishing returns to only increasing one of the inputs and not the other. In particular, if we take one of C xpm ⁢ ( t ) and L C ⁢ ( t ) to infinity but hold the other input fixed, then experiment throughput should remain bounded. Indeed, if you have infinite coding labor but only a fixed amount of experiment compute, you can only implement so many experiments per unit time; and if you have infinite experiment compute but only a fixed number of coders, you don’t have enough workers to code up experiments that (usefully) use the compute. This situation comes up in economics, where we would say that experiment compute and coding labor are complementary inputs that are hard to substitute for each other. It’s well modeled with a CES (“constant elasticity of substitution”) production function with substitution parameter ρ X < 0 :

X ⁢ ( t ) = ( α ⁢ C ~ xpm ⁢ ( t ) ρ X + ( 1 − α ) ⁢ L ~ C ⁢ ( t ) ρ X ) 1 / ρ X , 0 < α < 1 , ρ X < 0 ,

where

C ~ xpm ⁢ ( t ) = C xpm ⁢ ( t ) ζ , L ~ C ⁢ ( t ) = L C ⁢ ( t ) λ , 0 < ζ < 1 ,  0 < λ < 1 .

What is ρ ? The smaller the substitution parameter ρ is, the less it’s possible to only increase one of C xpm ⁢ ( t ) and L C ⁢ ( t ) in order to increase X ⁢ ( t ) .

What is α ? The closer 0 < α < 1 is to 1 , the more experiment compute is considered important relative to coding labor.

What is λ ? We use the smaller quantity L C ⁢ ( t ) λ instead of L C ⁢ ( t ) since L C ⁢ ( t ) is a measure of parallel coding labor but we want a measure of serial coding labor (a single programmer can make more progress in two days than a team of two programmers can make in a day because of the “stepping on toes” effect). The reason we use serial (rather than parallel) coding labor in the CES function is that if the project had unlimited experiment compute, the way we double the experiment throughput is by letting the coding workforce work for twice as long, rather than doubling the size of the coding workforce.

What is ζ ? The reason we discount experiment compute by the exponent 0 < ζ < 1 is that, if one had unlimited coding labor, doubling experiment compute doesn’t double experiment throughput because model size increases, meaning the ability to run near-frontier experiments goes down.

We set the parameter values as follows:

  1. 1.

    λ : We estimated this based on our intuitions about the effect of increased parallel labor on research effort, as well as interviews with frontier AI researchers.

  2. 2.

    ζ , α , ρ X : We estimated three quantities, from a fixed starting point of 2024, that together pin down the values of ζ , α , and ρ X : (a) the decrease in X ⁢ ( t ) from a 10 × decrease in C xpm , (b) the X ⁢ ( t ) asymptote as we take C xpm to infinity, and (c) the X ⁢ ( t ) asymptote as we take L C ⁢ ( t ) to infinity. To estimate these quantities, we combined frontier AI researcher interviews and our own reasoning about how X ⁢ ( t ) would change given infinite coding labor or compute.

Research effort, R ⁢ E ⁢ ( t ) , is a measure of how fast the AI project is going. It is the product of experiment throughput X ⁢ ( t ) and the aggregate experiment selection skill T ⁢ ( t ) :

R ⁢ E ⁢ ( t ) = T ⁢ ( t ) ⁢ X ⁢ ( t ) .

Doubling research effort means you get to any target capability in half the time. But research effort does not directly describe the fruits of the research, that is, how quickly software efficiency improves. To convert research effort into software improvements, we use the so-called “software law of motion” (see the Software Efficiency section).

Define the research stock, R ⁢ S ⁢ ( t ) , to be the accumulation of all research effort up through time t . In symbols,

R ⁢ S ⁢ ( t ) = ∫ − ∞ t R ⁢ E ⁢ ( τ ) ⁢ 𝑑 τ .

So research stock is to research effort what distance traveled is to speed. One can think of R ⁢ S ⁢ ( t ) as measuring the total amount of AI work the project has made, measured in units of “effort” or “investment” rather than “fruits” or “returns” of this effort.

Like research stock, S ⁢ ( t ) should be thought of as an operationalization of the total amount of AI progress made (where we abstract away hardware progress), this time focused on fruits/returns rather than effort/investment. As such, it’s natural to model S ⁢ ( t ) as a function of R ⁢ S ⁢ ( t ) :

R ⁢ S ≈  total AI software R&D work done → relationship? S ≈  total returns

When we look at past empirical data [cite], we see that S ⁢ ( t ) has been growing exponentially over time. Meanwhile, the numbers of AI researchers, of AI research engineers, of AI experiments run, and of papers submitted to machine learning conferences have all been growing exponentially over time as well. This suggests R ⁢ S ⁢ ( t ) has also been growing exponentially. So, since S ⁢ ( t ) and R ⁢ S ⁢ ( t ) both grow exponentially over time (with different bases for the exponentials), the relationship between them is best modeled by a power law:

S ⁢ ( t ) = C ⁢ R ⁢ S ⁢ ( t ) r , for some constant  C .

Intuitively:

each doubling of research stock  R ⁢ S → produces r  doublings of software efficiency  S

Note about diminishing returns: AI capabilities come from a combination of software efficiency S and training compute C train . Since training compute has been growing exponentially, for S to “keep up” and not fade into irrelevance, it also needs to be growing exponentially. So it’s natural to talk about returns on research effort in terms of S ¯ = log ⁡ S (i.e. orders of magnitude of S ). Since

S ¯ ≈ r ⁢ log ⁡ R ⁢ S ,

successive increases in R ⁢ S lead to smaller and smaller increases in S ¯ over time. This is the phenomenon of “diminishing returns”: the more progress has already been done, the harder it becomes to make impactful new discoveries since the low-hanging fruits have been picked.

Connection to the Jones semi-endogenous growth model

Jones 1995 uses the following growth law to model idea production:

A ′ ⁢ ( t ) = a ⁢ A ⁢ ( t ) 1 − 1 r ⁢ R ⁢ ( t ) λ .

Here A ⁢ ( t ) is the stock of ideas and R ⁢ ( t ) is the number of researchers engaged in idea production. The parameter λ > 0 controls the “stepping on toes” effect from adding additional researchers, and r > 0 controls the strength of the “fishing out” effect, according to which ideas become harder to find as the stock grows.

for various definitions of research effort R ⁢ E ⁢ ( t ) .

5.2 Effective compute

Effective compute, E ⁢ ( t ) , is the product of training compute C train ⁢ ( t ) and software efficiency S ⁢ ( t ) :

E ⁢ ( t ) = C train ⁢ ( t ) ⁢ S ⁢ ( t ) .

It’s a convenient capability measure that lets us attribute progress to both compute scaling and software improvements. The coding and experiment selection capabilities of AIs are expressed as functions of effective compute.

6.1 Automated coder (via time horizon)

An “automated coder” (or AC) is a specific capability milestone that influences the progression of the coding automation fraction, the coding automation efficiency, and the automated experiment selection skill (see those sections). An AC is defined as a collective of AIs that, if dropped into the present day, would be as productive on their own as all present human coders with no AI assistance.

We forecast the arrival date of AC by considering time horizons, which were introduced by METR in [cite METR study] to measure AIs’ coding ability. The coding time horizon, H ⁢ ( t ) , is the length of the coding tasks the frontier model can successfully complete 80% of the time, as measured by how long it takes humans to complete them. For example, if an AI has a 2 hour 80% time horizon, that means it can complete with an 80% success rate tasks that take human baseliners 2 hours (and for tasks that take baseliners more time, it has a lower success rate). We define our time horizon based on an imagined extension of the METR-HRS dataset, which we name METR-HRS-Extended. We use METR’s time horizon benchmark because (a) it’s a coding benchmark (b) that has an established trend from effective compute to capabilities and (c) the trend can at least in theory be extrapolated to very high levels of performance.

Defining our theoretical extended benchmark METR-HRS-Extended. Given that METR-HRS thus far only contains tasks that take their baseliners ≤ 30 hours and we want to forecast up to much higher levels, we need to define a procedure that could in principle be used to extend the task suite, though it would not be practical to actually do so indefinitely. It’s important to define which human baseliners are used during the extension: the more competent they are, the larger an impact we should expect at any given time horizon. We assume that METR finds human baseliners who take the same amount of time to complete tasks as a typical AGI company programmer who does similar tasks. We think that this might be loosely consistent with their past baselining, and this also represents a simple assumption that translates relatively well to real-world impacts.

We will primarily track what horizons we would expect to actually observe under our specified extension procedure, but we also assume for ease of reasoning about high reliability levels that there are no ambiguities or bugs in the tasks such that they could always be completed by a sufficiently competent agent. We refer to our theoretical extended and modified version of the task suite METR-HRS-Extended.

We focus on 80% time horizons, though METR also reports 50%, because we think higher reliability levels are more relevant. We also require AIs to achieve tasks efficiently in order for it to count as a success. In particular, we impose a constraint that the AI must accomplish tasks as efficiently as human researchers (see the Coding Automation Fraction section for more details on what we mean by “efficient automation”).

Why only 80% reliability? At the moment, we only support reliability levels of 80% since this is the highest reliability data that METR considers high-quality. How is it possible that an 80% time horizon could correspond to dominating the average coder across all categories of tasks? (a) There is some randomness in whether an AI succeeds at an individual task, so even an AI that dominates average humans across all categories will not have 100% reliability at human-level or above efficiency on individual tasks. (b) We adjust the required 80% time horizon up to account for increased reliability requirements.

Expandable: Why the efficiency threshold? The METR paper does not set an efficiency threshold, but we consider it necessary to set some sort of efficiency threshold, given that AIs are already sometimes less efficient than humans, and we expect this to become more important over time as AIs are better at translating more resources into better performance.

Although the original METR study emphasized the evolution of time horizon as a function of time t , we want to express time horizon as a function of effective compute. The benefit of doing so is that effective compute, unlike time, is a quantity that tracks the evolution of inputs to AI progress as well as the automation feedback loop. So while the “time horizon vs time” trend that METR observed may stop holding in the future, our assumption is that the “time horizon vs effective compute” trend will keep holding, so we can track time horizon by modeling effective compute.

Both time horizon and effective compute increased exponentially over the last 5 years, which suggests time horizon might grow exponentially as a function of OOMs (orders of magnitude) of effective compute. Writing E ¯ ⁢ ( t ) = log ⁡ ( E ⁢ ( t ) ) for OOMs of effective compute and t p for the present-day, a first pass is to express H ⁢ ( t ) as:

H ⁢ ( t ) = H ⁢ ( t p ) ⋅ 2 E ¯ ⁢ ( t ) − E ¯ ⁢ ( t p ) τ modified below

The parameter τ is the number of extra OOMs of effective compute needed to double time horizon:

effective compute increases by  τ  OOMS → produces 1 doubling of time horizon

We primarily set τ to fit historical METR-HRS trends, balancing between the long-term observed trend and a potential near-term uptick.

In order to accommodate potential changes in the rate of time horizon growth, we allow the relationship between time horizon and effective compute OOMS to be faster or slower than exponential. We might expect superexponential growth because of the “long-horizon agency argument”: as an AI learns to complete increasingly long tasks, it may actually develop better long-horizon agency skills (long-term planning, decomposing long tasks, noticing and correcting mistakes, etc.) than the METR-HRS-Extended human baseliners.

Superexponential time horizon growth possibilities At this point, there are 3 possibilities:

  1. 1.

    The success rate vs. time horizon trend “flips” to AIs having higher success rates at higher human-benchmarked time horizons, leading to an infinite/undefined 80% time horizon. This might happen because AIs are now differentially better at long-horizon tasks than humans, as opposed to short-horizon tasks; recall that the time horizons are defined relative to humans, as opposed to some inherent property of the task. In this case, our best guess is that we should interpret this as the METR-HRS-Extended time horizon being infinity in finite effective compute and that therefore we should expect superexponential time horizon growth leading up to this.

  2. 2.

    The success rate decreases as the human-benchmarked time horizon increases, but it asymptotes above 80% as we take the time horizon to infinity, resulting in an infinite horizon. Success rate may still decrease over time because the AI being better at short-horizon skills could outweigh being better at long-horizon skills. We’re not sure how to best think about which AGIs it would asymptote above 80%. Intuitively, for a very superhuman AGI it seems like it should, but it also seems a bit strange to assume that it would asymptote as opposed to continuing to decrease.

  3. 3.

    AIs seem to be better than humans at every task length we can measure, but they still in theory have a finite 80% time horizon. 6.2 Coding automation fraction and efficiency

6.2.1 Coding automation fraction

The coding automation fraction, A ⁢ ( t ) , is the fraction of coding work involved in frontier AI research that can be efficiently automated.

Clone-parity coders. We define a “clone-parity coder” by the condition that, if every human coder were replaced by a clone-parity coder, productivity would remain the same. For each type of coding task, we weight its importance by how many clone-parity coders would need to work on this task to not lose in productivity. We consider a task efficiently automated when AIs can complete the task faster than the human coders using no more than B N of automation compute (check that “automation compute” here also means for the baseline-date), where B is the number of human clone-parity-coder-equivalents working on the task and N is the total number of human coders (both for some baseline date).

For example, A ⁢ ( March 2029 ) = 80 % means that we could use the AIs of March 2029 to automate away 80 % of the coding labor-units done at the baseline-date and not lose in productivity ( 80 % as measured by the proportion of clone-parity-coders doing those labor-units).

6.2.2 Coding automation efficiency

We use indices i in the interval [ 0 , 1 ] to model the various units of coding labor needed to implement AI experiments. At any given time t , the labor-units i in [ 0 , A ⁢ ( t ) ] are considered efficiently automated, while the labor-units i in [ A ⁢ ( t ) , 1 ] are not.

For each labor-unit i , the coding automation efficiency, η i ⁢ ( t ) is the conversion rate between units of automation compute and human coders. So, if C aut , i ⁢ ( t ) is the amount of compute assigned to labor-unit i at time t , then this corresponds to η i ⁢ ( t ) ⁢ C aut , i ⁢ ( t ) human coders.

Writing E i for the effective compute corresponding to when task i is first efficiently automated, we use

η i ⁢ ( t ) = { η init ⋅ ( E ⁢ ( t ) E i ) η slope if task  i  is efficiently automated by time  t ; 0 otherwise;

where η init is a parameter representing the initial efficiency when any task first gets automated, and η slope is a parameter controlling how quickly automated coders’ efficiency continues to improve afterward. This can be intuitively understood as:

each doubling of effective compute  E → produces η slope  doublings of coding automation efficiency

These efficiency gains come from a combination of making AIs faster, being able to run more AIs, and getting more capable AIs that can complete tasks with fewer actions. (By our definition of when a coding labor-unit is efficiently automated, the initial conversion rate η init is the ratio of the total number of human coders to the amount of automation compute in the baseline year.)

6.3 Automated experiment selection skill

Experiment selection skill measures the value one gets per experiment. We modeled human experiment selection skill with a lognormal distribution Lognormal ⁢ ( μ , σ 2 ) (see the section about human experiment selection skill for details).

We describe the AIs’ experiment selection skill in terms of standard deviations σ of the (underlying normal of the) human distribution. Write T A ⁢ I S ⁢ D ⁢ ( t ) for the number of human standard deviations from the human median (if T A ⁢ I S ⁢ D ⁢ ( t ) is negative, this means the AIs’ experiment selection skill is below the human median). We assume the AIs’ experiment selection skill increases by a fixed rate T rate for each additional OOM of effective compute:

T A ⁢ I S ⁢ D ⁢ ( t ) = T A ⁢ C + T rate ⁢ ( E ¯ ⁢ ( t ) − E ¯ A ⁢ C ) in human standard deviations  σ .

Here T A ⁢ C is a model parameter for the experiment selection skill of the AIs at the coding automation anchor (see the Automated Coder section).

Moving from standard deviations to absolute units, the AIs’ experiment selection skill is

T A ⁢ I ⁢ ( t ) = exp ⁡ ( μ + σ ⁢ T A ⁢ I S ⁢ D ⁢ ( t ) ) .
